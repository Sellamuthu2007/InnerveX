======================================================================================
                            INNERVEX DIGIBANK PROJECT
                     Digital Certificate Management Platform
======================================================================================

PROJECT OVERVIEW
----------------
InnerveX DigiBank is a comprehensive blockchain-based digital certificate management 
platform that enables institutions to issue, manage, and verify digital certificates 
with tamper-proof security. The system provides a complete ecosystem for certificate 
lifecycle management, from issuance to verification, sharing, and revocation.

The platform serves as a "Digital Certificate Bank" where educational institutions, 
employers, and regulatory bodies can securely manage digital credentials, while 
individuals can store, share, and prove ownership of their certificates.


======================================================================================
CORE FEATURES
======================================================================================

1. DIGITAL CERTIFICATE ISSUANCE
   - Institutions can issue digital certificates to students/recipients
   - Support for PDF, PNG, JPEG certificate uploads
   - Automatic certificate hashing using SHA-256 for tamper detection
   - Blockchain anchoring for immutability (simulated blockchain)
   - Email notifications sent to recipients upon issuance
   - QR code generation for quick verification
   - Certificate metadata support (expiry dates, custom fields)

2. CERTIFICATE REQUEST SYSTEM
   - Individuals can request certificates from institutions
   - Institution dashboard to approve/reject requests
   - Workflow management for certificate requests
   - Rejection reason tracking
   - Email notifications for request status updates

3. CERTIFICATE SHARING
   - Secure certificate sharing via unique tokens
   - Time-limited access to shared certificates
   - Access count tracking and audit logs
   - Email sharing with recipient verification
   - IP address and user agent tracking for security
   - Shareable link generation for third-party verification

4. BLOCKCHAIN INTEGRATION (SIMULATED)
   - Certificate anchoring to blockchain ledger
   - Transaction hash generation for each certificate
   - Block number tracking
   - Gas usage simulation
   - Wallet address management for issuers
   - Support for multiple networks (Polygon, Ethereum, BSC)
   - Explorer URL generation for blockchain transactions

5. CERTIFICATE VERIFICATION
   - Public verification via certificate ID or QR code
   - Real-time verification status checking
   - Certificate hash validation
   - Blockchain transaction verification
   - Issuer and recipient validation
   - Expiry date checking

6. CERTIFICATE REVOCATION
   - Institutions can revoke issued certificates
   - Revocation reason documentation
   - Blockchain transaction for revocation
   - Email notifications for revoked certificates
   - Audit trail for revocation actions

7. MULTI-ROLE USER SYSTEM
   - Individual: Receive, view, and share certificates
   - Institution: Issue, manage, and revoke certificates
   - Employer: Verify employee certificates
   - Regulatory: Oversight and compliance monitoring

8. ADVANCED FEATURES
   - Dual-consent verification for sensitive certificates
   - User reputation system based on certificate history
   - Fraud detection and management
   - Institution trust registry
   - Certificate lifecycle tracking
   - Activity timeline for all actions
   - Certificate balance dashboard

9. NOTIFICATION SYSTEM
   - Real-time notifications for certificate events
   - Email notifications for important actions
   - In-app notification center
   - Notification preferences management

10. SECURITY FEATURES
    - JWT-based authentication
    - Password hashing with bcryptjs
    - Rate limiting to prevent abuse
    - CORS configuration for API security
    - NoSQL injection prevention
    - Input sanitization and validation
    - File upload validation
    - Helmet.js security headers

======================================================================================
DATABASE MODELS
======================================================================================

1. USER MODEL (User.js)
   Fields:
   - name: User's full name (required)
   - email: Unique email address (required, indexed)
   - password: Hashed password (bcrypt, min 8 chars)
   - role: [individual, institution, employer, regulatory]
   - walletId: Blockchain wallet identifier (unique)
   - isVerified: Account verification status
   - isActive: Account active status
   - certificatesIssued: Counter for issued certificates
   - certificatesReceived: Counter for received certificates
   - lastLogin: Last login timestamp
   - timestamps: createdAt, updatedAt

2. CERTIFICATE MODEL (Certificate.js)
   Fields:
   - title: Certificate title/name
   - issuerName: Name of issuing institution
   - issuerId: Reference to User (institution)
   - recipientName: Name of recipient
   - recipientEmail: Recipient email address
   - recipientId: Reference to User (recipient)
   - status: [verified, pending, revoked]
   - fileData: Base64 encoded certificate file
   - fileName: Original file name
   - fileType: MIME type [pdf, png, jpeg, jpg]
   - fileSize: File size in bytes
   - certificateHash: SHA-256 hash for tamper detection
   - revocationReason: Reason if revoked
   - revokedAt: Revocation timestamp
   - revokedBy: Reference to User who revoked
   - expiresAt: Certificate expiry date
   - metadata: Additional custom fields (Map)
   - timestamps: createdAt, updatedAt

3. REQUEST MODEL (Request.js)
   Fields:
   - title: Requested certificate title
   - institutionName: Target institution name
   - institutionId: Reference to User (institution)
   - recipientName: Requester name
   - recipientEmail: Requester email
   - recipientId: Reference to User (requester)
   - status: [sent, approved, rejected]
   - rejectionReason: Reason if rejected
   - processedAt: Processing timestamp
   - processedBy: Reference to User who processed
   - timestamps: createdAt, updatedAt

4. SHARE MODEL (Share.js)
   Fields:
   - certificateId: Reference to Certificate
   - recipientEmail: Email of person certificate is shared with
   - recipientId: Reference to User (if registered)
   - sharedByEmail: Email of person sharing
   - sharedById: Reference to User (sharer)
   - shareToken: Unique token for access (UUID)
   - expiresAt: Share link expiry date
   - accessCount: Number of times accessed
   - lastAccessedAt: Last access timestamp
   - accessLog: Array of access records (IP, user agent, timestamp)
   - timestamps: createdAt, updatedAt

5. BLOCKCHAIN LEDGER MODEL (BlockchainLedger.js)
   Fields:
   - certificateId: Reference to Certificate
   - certificateHash: SHA-256 hash
   - txHash: Transaction hash (simulated, 0x + 64 hex chars)
   - blockNumber: Block number in blockchain
   - issuerWallet: Wallet address (0x + 40 hex chars)
   - network: [Polygon Mainnet, Ethereum Mainnet, BSC Mainnet]
   - status: [PENDING, CONFIRMED, FAILED]
   - gasUsed: Gas consumed (21000-500000)
   - anchoredAt: Anchoring timestamp
   - revoked: Revocation status
   - revocationTxHash: Revocation transaction hash
   - revokedAt: Revocation timestamp
   - timestamps: createdAt, updatedAt

6. NOTIFICATION MODEL (Notification.js)
   Fields:
   - userId: Reference to User
   - title: Notification title
   - message: Notification content
   - type: [info, success, warning, error]
   - read: Read status
   - link: Optional action link
   - timestamps: createdAt, updatedAt

7. AUDIT LOG MODEL (AuditLog.js)
   Fields:
   - userId: Reference to User performing action
   - action: Action performed
   - entityType: Type of entity affected
   - entityId: ID of affected entity
   - details: Additional action details
   - ipAddress: User's IP address
   - userAgent: User's browser/client info
   - timestamps: createdAt

======================================================================================
USE CASES AND WORKFLOWS
======================================================================================

1. CERTIFICATE ISSUANCE WORKFLOW
   a) Institution logs in
   b) Navigates to "Issue Certificate" page
   c) Fills in recipient details (name, email)
   d) Uploads certificate file (PDF/Image)
   e) Adds metadata (expiry date, achievement details)
   f) Submits the form
   g) System generates certificate hash
   h) Certificate is anchored to blockchain
   i) Email notification sent to recipient
   j) Certificate appears in institution's "Issued" list
   k) Certificate appears in recipient's "My Certificates"

2. CERTIFICATE REQUEST WORKFLOW
   a) Individual logs in
   b) Navigates to "Request Certificate" page
   c) Fills in institution name and certificate details
   d) Submits the request
   e) Request appears in institution's "Pending Requests"
   f) Institution reviews the request
   g) Institution approves or rejects with reason
   h) Email notification sent to requester
   i) If approved, certificate is created and issued
   j) If rejected, request is marked as rejected

3. CERTIFICATE SHARING WORKFLOW
   a) Certificate owner logs in
   b) Navigates to specific certificate
   c) Clicks "Share Certificate" button
   d) Enters recipient email address
   e) Optionally sets expiry date for share link
   f) Submits the share request
   g) System generates unique share token
   h) Email sent to recipient with share link
   i) Recipient clicks link (no login required)
   j) Recipient views certificate details
   k) Access is logged with IP and timestamp

4. CERTIFICATE VERIFICATION WORKFLOW
   a) Verifier visits public verification page
   b) Enters certificate ID or scans QR code
   c) System looks up certificate
   d) System validates:
      - Certificate exists
      - Not revoked
      - Not expired
      - Hash matches
      - Blockchain record exists
   e) Verification result displayed with:
      - Issuer details
      - Recipient details
      - Issue date
      - Blockchain transaction
      - Status badge

5. CERTIFICATE REVOCATION WORKFLOW
   a) Institution logs in
   b) Navigates to certificate details
   c) Clicks "Revoke Certificate" button
   d) Enters revocation reason
   e) Confirms revocation
   f) Certificate status changed to "revoked"
   g) Revocation recorded on blockchain
   h) Email notification sent to recipient
   i) Certificate marked as invalid in all verifications

CONCLUSION
======================================================================================

InnerveX DigiBank is a comprehensive, modern, and secure digital certificate 
management platform that bridges the gap between traditional paper certificates 
and blockchain-based digital credentials. 

The platform provides a complete ecosystem for certificate lifecycle management, 
from issuance to verification, with features like blockchain anchoring, secure 
sharing, public verification, and comprehensive audit trails.

Built with modern web technologies (React, Node.js, MongoDB) and containerized 
with Docker, the platform is scalable, maintainable, and ready for production 
deployment.

The simulated blockchain layer provides the foundation for future integration 
with real blockchain networks, making it a forward-compatible solution for 
digital credential management.
